# Technical SEO Findings — pinexadigital.com
**Score: 24/100** | Audit date: 2026-07-04 | Previous audit: 2026-06-29 (score 28/100)

## Summary

Since the 2026-06-29 audit, several code-level fixes have genuinely landed: the sitewide `noindex` in `layout.tsx` is gone (now `robots: { index: true, follow: true }`), the homepage exports proper metadata, the six blog post detail pages now return 200 instead of 404, the `/api/contact` route exists, and a `next.config.ts` `headers()` block now ships four of five standard security headers. **However, the single most damaging issue — `robots.txt` disallowing the entire site (`Disallow: /`) — is UNCHANGED and UNRESOLVED 5 days after being flagged Critical.** It is generated by `src/app/robots.ts`, which still hardcodes `rules: { userAgent: "*", disallow: "/" }`. This one file overrides every other fix: Googlebot, Bingbot, and all AI crawlers are still instructed to crawl nothing, so the corrected meta robots tags, the working blog pages, and the new security headers currently have zero effect on organic visibility. A new High-severity issue also surfaced this cycle: canonical tags, sitemap URLs, and Open Graph URLs all resolve to the bare apex `https://pinexadigital.com` (no `www`), but the site's live/served host is `https://www.pinexadigital.com` — meaning every canonical tag on the site points at a URL that itself 308-redirects elsewhere, which is a self-defeating canonical signal.

## Critical

### 1. robots.txt still disallows the entire site — unresolved regression (5 days)
**File:** `src/app/robots.ts`

Confirmed live at `https://www.pinexadigital.com/robots.txt`:
```
User-Agent: *
Disallow: /

Sitemap: https://pinexadigital.com/sitemap.xml
```
Source, unchanged since the prior audit:
```ts
export default function robots(): MetadataRoute.Robots {
  return {
    rules: { userAgent: "*", disallow: "/" },
    sitemap: siteUrl("/sitemap.xml"),
  };
}
```
This blocks Googlebot, Bingbot, GPTBot, ClaudeBot, PerplexityBot, and every other compliant crawler from fetching any URL on the site. It fully nullifies the meta-robots fix, the now-working blog pages, and the new security headers — none of it matters if crawlers are told not to enter. This was flagged Critical on 2026-06-29 and is still live today, 2026-07-04.

**Recommendation:** Change line 6 to `rules: { userAgent: "*", allow: "/" }` (or remove the `rules` override entirely and let it default to allow). Redeploy, then re-verify with `curl https://www.pinexadigital.com/robots.txt` before closing this item. Submit the site for re-crawl in Google Search Console and Bing Webmaster Tools immediately after the fix ships, since the site has likely accumulated zero indexed pages during this outage.

## High

### 2. Canonical tags, sitemap, and Open Graph URLs all use the wrong host (apex, not www)
**Files:** `src/lib/site-config.ts` (`rootDomain: "pinexadigital.com"`, used by `siteUrl()`), every page's `alternates.canonical`, `src/app/sitemap.ts`, `src/app/layout.tsx` (OG/Twitter `url`/`images`)

The live site serves at `https://www.pinexadigital.com` (apex 308-redirects to www on every hit, confirmed on `/`, `/services`, `/pricing`, etc.). Yet every canonical tag site-wide resolves to the non-www apex, e.g. on `/services/seo`:
```html
<link rel="canonical" href="https://pinexadigital.com/services/seo"/>
```
Same pattern confirmed on all 11 sitemap pages, `/about`, `/contact`, `/blog`, `/privacy`, and `/terms`. `sitemap.xml` also lists apex URLs (`<loc>https://pinexadigital.com/services</loc>`), and OG/Twitter meta (`og:url`, `og:image`) use the apex host too.

This means every canonical tag points to a URL that itself returns a 308 redirect rather than the final resolved page — the canonical should point at the actual served URL. Search engines generally resolve this correctly (redirect target usually wins as canonical), but it is fragile, wastes a redirect hop for anything that respects the canonical literally (social crawlers, some AI crawlers), and is an easy, no-excuse fix since it's centralized in one config value.

**Recommendation:** Change `rootDomain` in `src/lib/site-config.ts` to `"www.pinexadigital.com"` (or add a distinct `canonicalHost` constant if apex must stay the brand-facing domain elsewhere). Regenerate `sitemap.xml` and redeploy. Confirm afterward that `curl -s https://www.pinexadigital.com/ | grep canonical` shows the `www` host.

## Medium

### 3. Duplicate brand name in `<title>` on two pages
**Files:** `src/app/services/seo/page.tsx:8`, `src/app/services/maintenance/page.tsx:8`

Both pages hardcode `"| PinexaDigital"` inside their own `title` string, and the root layout's title template (`template: "%s | PinexaDigital"` in `layout.tsx:27`) appends it again. Live result:
```html
<title>SEO Services for US Businesses | PinexaDigital | PinexaDigital</title>
<title>Website Maintenance &amp; Support Plans | PinexaDigital | PinexaDigital</title>
```
Confirmed via direct fetch of both pages. This wastes SERP title pixel width, looks unpolished in search results, and dilutes keyword relevance in the title tag.

**Recommendation:** Remove the trailing `| PinexaDigital` from the `title` string in both files — let the layout template add the brand suffix once, matching every other page (e.g. `/about`, `/pricing`, `/services/web-design` already do this correctly).

### 4. No Content-Security-Policy header
**File:** `next.config.ts`

`securityHeaders` array ships `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Strict-Transport-Security`, and `Permissions-Policy` — but no `Content-Security-Policy`. Confirmed absent via header dump on `/`, `/services`, `/pricing`, `/portfolio`, `/about`, `/contact`, `/blog`. This is a real improvement over the prior audit (which had none of the five), but CSP is the one still missing, leaving XSS/injection mitigation weaker than it should be.

**Recommendation:** Add a CSP header, starting permissive and tightening iteratively (Next.js apps typically need `'unsafe-inline'` for inline hydration scripts unless nonces are wired up):
```ts
{ key: "Content-Security-Policy", value: "default-src 'self'; img-src 'self' data: https://images.unsplash.com; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; font-src 'self' data:;" }
```
Test thoroughly in a staging deploy before production — Next.js inline JSON-LD `<script>` tags and hydration scripts commonly break under a strict CSP without `'unsafe-inline'` or nonces.

### 5. Sitemap.xml omits all 5 blog post detail pages
**File:** `src/app/sitemap.ts`

The route list in `sitemap.ts` is a hardcoded array of 11 top-level paths and does not include `/blog/how-much-does-a-website-cost`, `/blog/seo-for-small-business-us`, `/blog/shopify-vs-woocommerce-2026`, `/blog/website-speed-optimization`, or `/blog/web-design-trends-us-2026` — all five confirmed live (HTTP 200) and internally linked from `/blog`. These pages exist and are crawlable but rely entirely on internal-link discovery rather than sitemap declaration, which slows initial indexing and gives them no explicit priority/changefreq signal.

**Recommendation:** Generate blog post entries dynamically (e.g. loop over the same data source that powers `/blog` and `blog/[slug]/page.tsx`) instead of a static array, so new posts are automatically included.

### 6. Sitemap `lastModified` uses request-time `new Date()`, not real content dates
**File:** `src/app/sitemap.ts:21`

`lastModified: new Date()` sets every URL's lastmod to the moment the sitemap is generated/requested, not the actual last edit date of that page's content. Confirmed live: all 11 URLs in `sitemap.xml` share the identical timestamp `2026-07-04T11:50:19.79Z`. This gives crawlers a false "everything just changed" freshness signal on every rebuild/deploy and undermines the `changeFrequency` hints already configured per-route.

**Recommendation:** Track and pass a real per-page last-modified date (e.g. from CMS/frontmatter/git blame for blog posts, or a static constant for rarely-changed pages like `/about`), and only bump it when content actually changes.

## Low

### 7. og-image.jpg still returns 404
Confirmed `https://www.pinexadigital.com/og-image.jpg` → 404. `metadataBase` + `openGraph.images` + `twitter.images` in `layout.tsx` all reference `/og-image.jpg`, so every social share (LinkedIn, X/Twitter, Slack, Google Discover) will render without a preview image. This was flagged High in the prior audit and remains open; downgraded to Low here since it does not affect crawlability or indexing, only social-share presentation.

**Recommendation:** Add a 1200×630 branded JPEG at `public/og-image.jpg`.

### 8. Double redirect hop for the insecure apex URL
`http://pinexadigital.com/` (bare HTTP, no www) takes 2 hops to reach the final URL: `http://pinexadigital.com` → `https://pinexadigital.com` (308) → `https://www.pinexadigital.com` (308). All other combinations (`https://pinexadigital.com`, `http://www.pinexadigital.com`) are a clean single 308 hop to `https://www.pinexadigital.com`. Minor, since almost no real traffic enters via bare-HTTP-apex, but it is measurable latency/PageRank-dilution for any link built to that exact form.

**Recommendation:** Configure the redirect layer (Vercel) to combine host and scheme normalization into one hop for the http+apex case, or add an explicit `next.config.ts` redirect rule.

### 9. No IndexNow implementation
No IndexNow key file found (`/[key].txt` pattern, or any `indexnow.txt`), and no code references to the IndexNow API. This is optional but low-effort — Bing, Yandex, and Naver support near-instant reindexing via IndexNow, which would help re-establish visibility quickly once the robots.txt Critical fix ships.

**Recommendation:** Generate an IndexNow key, host it at `/{key}.txt`, and call the IndexNow API (`https://api.indexnow.org/indexnow`) on deploy/content-change via a small script or the `indexnow_submit.py` helper.

### 10. `/privacy` and `/terms` exist but are not in sitemap.xml
Both pages return 200 and have their own canonical tags, but are absent from the 11-URL sitemap. Likely intentional (legal boilerplate, low priority) — flagged Info-level for completeness, not necessarily a defect.

**Recommendation:** No action required unless these pages target any search intent; otherwise leave as discoverable-but-unlisted.
